# Data model decisions (the “why”)

This document explains **why the schema looks the way it does**. It’s written for a mixed audience: non-technical stakeholders who want the rationale, and engineers who need the concrete modeling choices.

## 1) Why a relational database (PostgreSQL vs NoSQL)

The referral program manages subscription state, eligibility rules, and one-time redemption of passes — all of which require **strong consistency**.

**Decision**: use **PostgreSQL** (relational).

Key reasons (from the design trade-offs):

- **ACID transactions**: redemption must be all-or-nothing (create trial + mark pass redeemed).
- **Subscription state consistency**: “active vs trial vs cancelled” must never drift.
- **Join-heavy queries**: passes belong to subscriptions and users; relational joins are a first-class feature.
- **Auditability**: foreign keys and constraints make audit trails safer and easier to trust.

The design explicitly acknowledges the trade-off: PostgreSQL requires migrations and schema discipline, but those are acceptable costs for financial/subscription correctness.

## 2) Table-by-table design decisions

### `users`

#### Why dual email columns (`email` + `email_normalized`)

- **Problem**: many email providers support aliases that map to the same inbox (e.g. Gmail `+` aliases, Gmail dot-ignoring). Without normalization, one person can create many “unique” accounts and claim unlimited trials.
- **Decision**: store both:
  - `email`: what the user typed (display/login identifier)
  - `email_normalized`: a provider-aware canonical form used for abuse prevention and uniqueness
- **Why it’s effective**: the system can enforce **one trial per real inbox** by placing a unique constraint/index on `email_normalized`.

This is a v2 security hardening decision.

---

### `subscriptions`

#### Why a unified subscriptions table (not a separate trials table)

The schema models **trials and paid subscriptions in the same table**.

**Decision**: one `subscriptions` table that contains both trial and paid states.

Why:

- **Trial → paid conversion is simpler**: conversion can be modeled as updating a row rather than migrating data between tables.
- **Continuous history**: a user’s subscription lifecycle stays in one place.
- **Fewer moving parts**: avoids duplicating fields and logic across two tables.

The design also calls out “orthogonal concepts”:

- `status` models the current state (`active`, `trial`, `cancelled`, `expired`)
- `is_trial` models whether the subscription is a trial or paid

This yields a flexible, explicit model that can represent “expired trial” vs “cancelled paid” cleanly.

---

### `referral_passes`

#### Why individual pass records (not a counter)

It’s tempting to track “3 passes remaining” as a number, but that fails the product requirements.

**Decision**: store **one row per pass**.

Why individual rows:

- **Unique tokens**: each pass needs its own shareable token.
- **Single-use behavior**: one specific pass should become redeemed and auditable.
- **Expiration per pass**: passes can have their own `expires_at`.
- **Analytics**: the business can measure conversion per pass and time-to-redeem.
- **Audit trail**: the system can record who redeemed which pass.

The trade-off is storage, but the design estimates it’s negligible at realistic scale.

#### Why denormalize `user_id` in `referral_passes`

Even though `referral_passes.subscription_id → subscriptions.user_id` exists, the schema also stores `referral_passes.user_id`.

**Decision**: denormalize `user_id` for performance.

Why:

- Listing passes for a user is a hot path. Denormalizing avoids an extra join.
- Indexed lookups become simpler and faster (`WHERE referral_passes.user_id = $1`).

Trade-off:

- Additional storage and a data consistency burden.

The design accepts the trade-off because the performance gain is meaningful and the write path is controlled (passes are generated by a trigger and updated in a transaction).

---

### `subscription_history`

#### Why an append-only audit trail table

Subscription and trial state changes are business- and compliance-relevant.

**Decision**: maintain an immutable `subscription_history` log.

Why:

- Clear traceability for support and analytics (“when did this change and why?”).
- Reduced ambiguity when debugging redemption and expiration issues.
- Creates a reliable event stream of state transitions without depending on application logs.

## 3) Modeling decisions (cross-cutting)

### Trial representation (`is_trial` + `trial_end_date`)

Trials are represented as subscriptions with:

- `is_trial = TRUE`
- a required `trial_end_date`

Paid subscriptions have:

- `is_trial = FALSE`
- `trial_end_date = NULL`

This is enforced with a database check constraint to prevent invalid states.

### Pass tracking (individual tokens)

Each referral pass has a unique `token` (designed as a JWT in v2), which enables:

- Secure sharing via URLs
- Built-in token expiration (JWT `exp`) plus a database expiration (`expires_at`)
- Stateless transport of “which pass is this” metadata, while still requiring database authorization checks

### Audit trail approach

Auditability is designed into the data model in two places:

- **Entity state**: pass row shows redemption fields (`is_redeemed`, `redeemed_by_user_id`, `redeemed_at`, `created_subscription_id`)
- **Lifecycle log**: `subscription_history` records subscription status changes over time

The redemption flow also explicitly writes a history record as part of the redemption transaction.

## 3b) Dual expiration: JWT `exp` vs `referral_passes.expires_at`

Both expire at 90 days by default, but they serve different purposes and can diverge:

| | JWT `exp` | `referral_passes.expires_at` |
|---|---|---|
| Enforced by | `jwt.verify()` (application) | Database check during redemption |
| Can be revoked early? | No (without a denylist) | Yes (admin UPDATE) |
| Can be extended? | No | Yes (admin UPDATE) |
| Fast check? | Yes (no DB query) | No (requires DB lookup) |
| Source of truth? | No | Yes |

**Typical flow** — both set to 90 days:
1. JWT check runs first (fast, no DB round-trip). Expired JWT → 410 Gone, no DB hit.
2. Database check runs second (authoritative). Expired `expires_at` → 410 Gone.

**Why keep both?**
- JWT expiration provides fast early rejection for clearly stale links.
- Database expiration allows the business to revoke or extend a pass independent of the JWT (e.g., re-issue a pass for a customer who lost their link).

**Note**: The current design sets both to 90 days and does not implement admin pass management. If those durations diverge in future, the database value is authoritative.

## 4) Trade-offs table (major decisions)

| Decision area | Options considered | Choice | Upside | Downside |
|---|---|---|---|---|
| Database type | PostgreSQL vs MongoDB | PostgreSQL | ACID transactions, strong consistency, joins, enforced constraints/auditability | Requires migrations and schema governance |
| Pass modeling | Counter vs individual rows | Individual rows | Unique tokens, per-pass expiration, detailed audit + analytics | More rows/storage (accepted as negligible) |
| Pass generation | Trigger vs application logic | Trigger | Atomic and impossible to “forget”; no extra round-trip | Harder debugging and testing (logic is “hidden”) |
| Token format | JWT vs UUID | JWT | Signed/tamper-proof, self-contained metadata, built-in expiration | Longer tokens, still requires DB lookups |
| Email identity | Raw email only vs normalized+raw | Dual columns | Prevents alias abuse; enforces uniqueness at inbox level | Adds complexity (normalization rules, migration) |
| Trial modeling | Separate trials table vs unified subscriptions | Unified subscriptions | Simplifies conversion + lifecycle; fewer tables | Requires careful constraints to prevent invalid trial fields |

---

## Summary

The v2 data model prioritizes correctness and abuse resistance:

- **PostgreSQL + transactions** for redemption correctness
- **Triggers and constraints** to make critical rules non-optional
- **A normalized email identity** to prevent infinite trial abuse
- **Explicit, auditable entities** (pass rows + subscription history) to support supportability and analytics
